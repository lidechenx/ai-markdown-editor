# 图片粘贴、原文展示与预览一致性问题（解决记录）

本文记录一次围绕「粘贴截图、引用式图片、CodeMirror 装饰、预览」的排查与定案过程，便于日后维护或复盘。

## 背景与现象

- **预览区**：`markdown-it` 渲染正常，图片可见。
- **原文区（CodeMirror）**：早期希望在源码里折叠长 Base64，曾用 `Decoration.replace` + `<img>` 做缩略图。
- **用户反馈**：
  1. 引用式写法 `![alt][id]` 与文末 `[id]: <data:...>` 在视觉上像「两段代码」，删除任一段都会导致预览从图片退化为纯文本或露出 Base64。
  2. 装饰层与「用法 + 定义」双段结构叠加时，容易出现**同一张图在心理上像重复出现**或编辑体验割裂。

## 根因说明（Markdown 语义）

**引用式图片**在 CommonMark / GFM 中由两部分组成，缺一不可：

1. 正文中的引用用法：`![说明文字][引用id]`
2. 文档中任意位置的引用定义：`[引用id]: <图片地址>`

因此「删掉第一段或第二段预览就坏」是**符合规范的预期行为**，不是渲染器随机故障。

## 方案演进（摘要）

1. **仅装饰定义行**：长 URL 折叠了，但 `![alt][id]` 仍是一大段可读性差的内容。
2. **同时为 `![alt][id]` 与定义行加装饰**：预览仍一份，但原文可能出现**两处**带图/占位，用户感知为重复。
3. **定义行紧凑、用法行缩略图**：依赖「是否检测到用法」的分支在极端文稿下仍可能不符合直觉。
4. **原文区统一改为「嵌入媒体芯片」**（SVG 图标 + 文字，**不**插入 `<img src="data:...">`）：避免在编辑器内重复解码 Base64，预览仍由 `markdown-it` 单独出图。
5. **粘贴策略定案（最终与「两段」问题一并解决）**：粘贴/拖拽/从 HTML 取图时，改为插入**单行**行内图：

   ```markdown
   ![文件名](<data:image/png;base64,...>)
   ```

   地址放在尖括号内，避免 data URL 中出现 `)` 时破坏 `](...)` 的配对。用户**一次删除一行**即可移除整张图，预览与源码一一对应。

## 涉及代码（维护入口）

| 模块 | 路径 | 职责 |
|------|------|------|
| 粘贴与拖拽 | `src/editor/imagePasteDrop.ts` | 插入 `![alt](<dataUrl>)`；文件去重 |
| 原文嵌入展示 | `src/editor/dataUrlImagePreview.ts` | 扫描行内 `![](...)` / `![](<...>)`、`![alt][id]`、`[id]: data/blob...`；芯片 Widget |
| 样式 | `src/style.css` | `.cm-md-embed-chip` 等 |
| 预览渲染 | `src/utils/renderMarkdown.ts` | `validateMdLink` 放行 `data:image/*` 等 |
| 预览图片尺寸 | `src/components/MdPreview.vue` | `max-width` / `max-height` / `object-fit` 约束大图 |

## 测试

- `src/editor/dataUrlImagePreview.test.ts`：扫描区间（含尖括号行内 data URL）。
- `src/utils/renderMarkdown.test.ts`：行内 `![](<data:...>)` 能输出 `<img>`。
- `src/editor/imagePasteDrop.test.ts`：粘贴去重逻辑。

运行：`npm run test`。

## 结论

- **旧文稿**中手写的引用式 `![x][id]` + `[id]: ...` 仍完全支持；装饰层仍会识别。
- **新粘贴**默认单行内嵌图，降低「两段必须一起维护」的心智负担。
- **原文区不出栅格图**、**预览区出图**，职责分离后，「重复图」与「双段误删」类问题从机制上收敛。
